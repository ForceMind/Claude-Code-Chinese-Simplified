## 二、缘起：为什么要自己做

### 痛点

1. **Claude Code 变成了原生二进制。** 早期 CC 是 Node 的 `cli.js`，改 JS 文本即可汉化。现在 CC 是 **Bun 编译的原生单文件可执行程序**（`claude.exe`，约 226MB，Windows 下是 PE 格式，Mac 下是 Mach-O），JS 包被嵌入在 exe 里。汉化 = **二进制补丁**，门槛陡增。

2. **现有方案 claude-code-zh-cn 跟不上了。** 参考项目 `claude-code-zh-cn`（作者 taekchef，v0.1.84）质量很高，但：
   - 更新缓慢，作者可能已弃坑；
   - 有 **support-window（支持窗口）** 机制，锁死在 `maxVersion 2.1.195`；
   - 一旦 CC 版本超出窗口，它就**拒绝工作**，界面退回英文。
   - 用户实测：**早已超出它的支持窗口**，且 CC Switch 更新后终端和切换器都退回英文。

3. **翻译滞后是本质矛盾。** CC 更新太频繁，人工维护的词典永远追不上新版本新增的英文文案。

### 我们的立场

> **本工具必须永久支持最新 CC，突破版本窗口限制。**

宁可翻译覆盖不到 100%，也要保证**任何新版本都能立刻用、不被锁死**。
（当前观测到的 CC 版本：**2.1.190**。）

## 三、探索过程：我们是怎么想清楚的

### 第 1 步：搞清楚 claude.exe 的内部结构

CC 是 Bun 编译产物。Bun 二进制的关键结构：

- 尾部魔术串 `BUN_TRAILER = "\n---- Bun! ----\n"`（用来识别是不是 Bun 二进制）；
- 偏移表（offsets）、字符串指针表（每个 8 字节，含长度）、编译模块、以及**嵌入的 JS 源码**。

### 第 2 步：读懂 claude-code-zh-cn 是怎么做的

逆向阅读了它的实现（在 `C:\Users\wxx11\.claude\plugins\claude-code-zh-cn\`）：

- `bun-binary-io.js`（LIEF 二进制 I/O 核心）——用 **LIEF 库**正式解析 PE/Mach-O，抽取嵌入的 JS 源，改完再**重打包**（`repackPE` / `repackMachO`），Mach-O 还要重新签名。
- `patch-cli.js`——补丁引擎：抽取 JS 源文本 → `applyTranslations(sourceText, map)` 做文本替换 → 重打包写回。
- `cli-translations.json`——文本词典，`[{en, zh}]` 格式，1890 条（我们把它作为上游词典基础）。

**它的方案本质：解析二进制结构 → 抽出 JS 文本 → 自由长度替换 → 重打包。** 强大，但**依赖 LIEF 原生库、流程复杂**。

### 第 3 步：关键的架构分岔

摆在面前两条路：

| 方案 | 优点 | 缺点 |
|---|---|---|
| **A. 等长原地字节替换**（零依赖） | 极简、无第三方依赖、任意版本可跑、真正可持续 | 中文字节数必须 ≤ 英文，超长的装不下 |
| **B. LIEF 解析+重打包**（照搬 zh-cn） | 翻译不受长度限制，可 100% 覆盖 | 依赖 LIEF 原生库、安装复杂、与"简单"冲突 |

### 决策：选 A（等长原地替换，零依赖）

**推理：**

1. **契合灵魂。** B 的重打包依赖 LIEF 原生库，安装复杂，直接违背"简单"。A 是一个无依赖的单文件脚本，才是真正的"可持续"（不会因依赖腐烂而失效）。

2. **A 的可行性论据（2026-07-10 实证修正）。** 早先以为"现役 exe 里有 800 处等长替换"，复核发现**不成立**——现役 claude.exe 比备份小 5246 字节，zh-cn 走的是重打包，长度会变。**真正成立的论据是**：词典里的中文 UTF-8 串确实存在于**正在运行的** claude.exe 内嵌 JS 中（多条实测 `zhInCur=true`），Bun 照常执行 → **Bun 运行时不校验内嵌 JS 完整性**。等长原地替换是重打包的严格子集（不动任何偏移表，更保守），安全性只强不弱。另有语法级验证兜底（见 §九）。

3. **A 天然永久支持新版。** 词典是 `en→zh` 的**文本对，与版本号无关**。只要新版 claude.exe 里还含那句英文，补丁就能命中。**新版新增的英文串**不在词典里 → 不硬报错、直接跳过 → 由 `scan` 报告出来供增量补充。这正好实现"永久支持 + 突破版本窗口"。

**关于覆盖率的取舍（诚实记录）：**

等长方案的代价是覆盖率。上游 1890 条词典里：
- **1177 条**中文字节 ≤ 英文，可直接用（等长安全）；
- **496 条**中文比英文长，等长装不下（暂存 `dict/oversize.json`，留给未来处理）；
- 217 条无效（无中文/非 ASCII 等）。

**等长覆盖率 ≈ 70.4%。** 我们接受这个 trade-off：

> **70% 中文界面 + 永久可用**，胜过 **100% 翻译但被锁死在旧版本**。

剩下 30% 由后续的增量机制和（可选的）重打包引擎逐步补齐。

## 四、核心机制设计

### 4.1 等长原地替换算法

对词典里每条 `en → zh`（已保证 `byteLen(zh) ≤ byteLen(en)`）：

1. 在二进制里搜索 `en` 的 ASCII 字节序列；
2. 命中处替换为 `zh 的 UTF-8 字节 + 0x20 空格填充`，凑满 `en` 的字节长度；
3. **字节长度不变** → 二进制结构不破坏 → 无需重打包。

词典按 `byteLen(en)` **降序**排列：先替长串，避免长串被短串的子串抢先替换而错位。

### 4.2 幂等 + 跟随新版（patcher 的关键设计）

**始终从"纯英文备份"出发打补丁，绝不在已汉化的产物上增量叠加。** 流程：

- 读 target，检测它当前是否**纯英文**（不含中文）：
  - **是纯英文**（新装/升级后的官方原版）→ 刷新备份 `copyFileSync(target, target+'.cchans-backup')`，以它为源。**这一步让工具自然跟随新版本。**
  - **已汉化**（我们或别人上次的产物）→ 用现有备份作纯英文源；找不到备份则拒绝并提示重装 CC。
- 从纯英文源做等长替换，写回 target。
- 断言：输出字节数 == 输入字节数（等长不变量）。

**硬链接注意：** CC 的 `install.cjs` 用 `linkSync`（硬链接）放置 claude.exe，所以备份必须用 `copyFileSync` 断开硬链接，避免备份和原文件是同一 inode。

### 4.3 scan：滞后检测 + 贡献/更新提醒（用户明确要求）

当 `scan` 发现"词典有、但此版 CC 里没有命中"或"CC 里有新的未译英文串"时，**按运行者身份给不同提示**：

- **维护者**（工具根目录存在 `.git`，即克隆的仓库）→
  提示："发现 N 处未覆盖，可在**本仓库启动 Claude Code**，让它翻译新串、更新 `dict/sources/cli-translations.json`，再跑 `build-dict`。"
- **普通用户**（无 `.git`，通过 npm 安装）→
  提示："你的 CC 版本可能比词典新。请**更新本工具**（`git pull` / `npm update`）获取最新词典；仍缺失欢迎去仓库提 issue/PR。"

用 `.git` 是否存在来区分身份，简单可靠。

## 九、2026-07-10 开发冲刺实录：顾问审阅 + 三个实证反转

按「先计划 → 问顾问 → 确认无问题 → 再开发」的流程走完了全程。顾问审阅 + 自主实证共发现三个**教科书式的反转**，全部用最低成本实验裁决，记录如下：

### 反转 1：等长方案的立论换了根基（见 §三 修正）

旧论据"现役 exe 有 800 处等长替换"被字节对比推翻（现役比备份小 5246 字节 = zh-cn 是重打包）。新论据更强：中文串实测活在运行中的内嵌 JS 里 → Bun 不校验 JS 完整性 → 等长是重打包的严格子集。

### 反转 2：`\uXXXX` 转义该保留，顾问建议反了

顾问认为词典里的字面 `—` 等转义是"抽取态污染"，应反解成真实 UTF-8 再匹配。**实测正相反**：二进制内嵌的是 JS 源码文本，源码里写的就是 `—` 这 6 个 ASCII 字符——8 条样本 7 条以字面形态直接命中，反解后全部打不中。**结论：保持原样。**（顾问的另一半意见是对的：`${...}` 占位符条目 30 条确实是死条目，且含混淆后的版本特定标识符，已剔除。）

### 反转 3：短词命中在标识符内部——顾问和我们都漏掉的 A 级风险

scan 的短词告警引出关键发现：`Sandbox` 434 处命中里 **301 处在压缩代码的标识符内部**（`SandboxNetworkAccess`、`queryCancelledError`、`supportsAdaptiveThinking:`）。等长替换的**填充空格会劈开标识符 → 语法错误 → 二进制报废**。zh-cn 的无填充替换侥幸免疫（CJK 是合法标识符字符，全局替换 = 一致改名）。修复双管齐下：

1. **patcher 词边界守卫**：needle 边缘字节是标识符字符时，命中位置相邻字节也是标识符字符则跳过（实测跳过 438 处）。
2. **build-dict 剔除裸标识符词**（`/^[A-Za-z][A-Za-z0-9_$]*$/`，11 条）：独立属性名 `{Sandbox:1}` 与被填充的字符串键 `"沙盒 "` 会不一致，整词剔除才彻底。

### 语法级黄金验证（新增的验证手段）

手工解析 PE section 表（40 字节/项，零依赖）→ 定位 `.bun` section → 按 `[size header][blob…trailer]` 结构抽出内嵌模块 → 主模块 `cli.js` 16.5MB。**补丁前后都用 esbuild 完整解析：均通过。** 注意 `node --check` 不可用（Node 22 不认 ES2026 的 `using` 语法，原版也过不了，属基线限制）。抽取脚本在开发 scratchpad，未进仓库（工具运行时不需要它）。

### 顾问其余结论的落实情况

- B1 性能（43s）：加进度提示，接受（每次升级跑一次，可忍）。
- B2 备份误判：已用通用 CJK 连续串检测（连续 ≥4 个 CJK 字符的串 ≥50 处判已汉化；实测纯英文原版仅 7 处、汉化版 200+，区分度充足），覆盖旧备份前先比对 + 保留 `.prev`。
- B3 Windows 占用：atomicWrite 捕获 EBUSY/EPERM 转友好提示。
- C 级伪风险（PE 签名/checksum/Bun 自校验/空格破坏字符串语义）：确认不拦截，不处理。

### 教训

1. **「继承来的证据」也要复核**——"800 处 sameSlot"是上一轮分析的结论，差点带着错误论据发布。
2. **顾问的价值在提出问题，裁决靠实验**——顾问指出词典污染方向完全正确，但具体修法（反解转义）反了；30 分钟实验避免了把 15% 死条目"修"成 100% 死条目。
3. **告警机制会反哺设计**——scan 的短词告警本来只是 B4 的"可观测性"需求，却直接钓出了最严重的 A 级 bug。
